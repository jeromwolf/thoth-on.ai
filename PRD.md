## 1. 개요 (Executive Summary)

보험 청구 데이터를 **개별 이벤트가 아닌 관계망(지식그래프)**으로 재구성하여, 룰·통계·단건 ML이 놓치는 **조직형·은밀형 사기**를 탐지하고, 조사관이 그 근거를 직관적으로 파악·소명할 수 있게 하는 플랫폼.

- **What:** Neo4j 기반 지식그래프 + 그래프 알고리즘(GDS) + 룰/ML 하이브리드 탐지 + **설명가능** 케이스 관리
- **Why now:** 적발액 고액화·조직화, 개정 특별법으로 외부 자료 연계 가능, 룰 단독 시스템의 한계 명확
- **For whom:** 손해보험사·생명보험사 SIU(특별조사팀), 공제조합, 데이터/리스크팀
- **Differentiator:** 탐지 점수가 아니라 **"왜 의심인지" 관계 경로를 자연어·시각화로 소명** (온톨로지 + LLM 설명 레이어)

## 2. 배경 및 문제 정의

### 2.1 시장 현황 (출처: 금융감독원 2025년 보험사기 적발현황, 2026.3)
- 2025년 적발 금액 **약 1조 1,571억 원** — 역대 최대, 4년 연속 1조 원 초과
- 적발 **인원 감소(약 10.6만 명)**, 금액 증가 → **건당 고액화**
- 종목별: **자동차보험 49.5%**, 장기보험(실손 포함) 39.8%
- 유형별: **사고내용 조작 54.9%**, 허위사고 20.2%, 고의사고 15.1%
- **병원·보험업 종사자 주도 조직적 사기 급증** (역할 분업형 범죄조직)

> 시사점: 단건·개인 단위 탐지로는 한계. **다수 주체(병원·정비소·브로커·계좌·차량)가 공유 엔티티로 얽힌 네트워크**를 봐야 함 → 그래프가 본질적으로 적합.

### 2.2 규제 환경
- **보험사기방지 특별법** 2016 제정, **개정법 2024.8.14 시행**
- 금융당국이 **건보공단·근로복지공단 등에 조사 자료 제공 요청** 가능 → 연계 데이터 소스 제도적 확대
- 사기 **알선·광고 금지** 신설 → 브로커/알선 네트워크 탐지 수요 증가
- **함의(제품 요건):** 개인정보·민감정보(의료) 처리 → 가명처리, 접근통제, 감사로그, **온프레미스/VPC 배포 필수**

### 2.3 기존 탐지 방식의 한계
| 방식 | 한계 |
|---|---|
| 룰 기반 | 알려진 유형만 식별, 신종·우회 패턴 취약 |
| 통계적 이상탐지 | 정상 변동을 오탐 처리 가능성 |
| ML(지도학습) | 정확도는 높으나 설명 어려움, 콜드스타트·편향 취약 |
| 네트워크 분석(단독) | 계산비용·전문성 요구 |

→ 공통 한계: 정적 규칙·수치 패턴에 의존해 **행위 간 숨은 의미·시맨틱 연결 구조를 포착하지 못함.**

### 2.4 기회
관계 중심 그래프로 **누가-어디서-무엇을-언제**를 통합 추적 → 조직형 네트워크 식별, 전이 경로 추적, **설명가능성** 확보. 레퍼런스: Zurich(노드 ~2천만·엣지 ~3천5백만), Intuit(Neo4j 보안 지식그래프).

## 3. 제품 비전 & 목표

### 3.1 비전
> "보험 청구의 단편적 이벤트를 의미 있는 관계망으로 전환하여, 보이지 않는 조직적 사기를 **빠르고 설명 가능하게** 차단한다."

### 3.2 목표 (Zurich 벤치마크 기준)
| 지표 | 베이스라인 | 목표 |
|---|---|---|
| 조사관 1인당 일일 의심청구 처리 | 현행 룰 기반 | **+10건 이상** |
| 건별 검토 시간 | 현행 | **10분 이상 단축** |
| 사기 적발률 | 룰 대비 | **3배 이상** |
| 탐지 결과 설명가능성 | 점수만 제공 | **관계 경로 + 자연어 근거 100%** |
| 오탐(FPR) | — | 룰 단독 대비 유의미 감소 (PoC 측정) |

### 3.3 Non-goals (이번 범위 제외)
- 보험금 지급 결제/정산 시스템 대체
- 차량/CCTV 영상 등 비정형 멀티모달 1차 처리 (외부 연동만)
- **실시간 트랜잭션 인라인 차단** — 본 제품은 **탐지·조사 지원**에 집중, 차단은 기간계 위임 (V2 "실시간 ingest"는 탐지 반영용이지 차단이 아님)

## 8. 기능 요구사항 (FR)
> **우선순위** P0(필수)/P1/P2 · 각 FR은 WP 단위이며 **AC**를 테스트 타깃으로 사용.

### 8.1 데이터 수집·적재 (Ingestion)
- **FR-1.1 (P0)** CSV/Parquet 배치 적재 파이프라인.
  - AC: 청구·고객·계약·차량·계좌·의료기관·정비소 소스를 노드/엣지로 매핑 적재, **멱등**(재실행 시 중복 0).
- **FR-1.2 (P0)** 엔티티 해소: 동일 주소/전화/계좌/차량 정규화·중복 병합. (Q4: 결정적 완전일치만 자동)
  - AC: 표기 차이(공백·하이픈) 정규화 후 동일 엔티티로 병합되는 테스트 통과.
- **FR-1.3 (P1)** 가명처리/마스킹(주민번호·연락처 해시).
  - AC: 원문 PII가 그래프에 평문 저장되지 않음.
- **FR-1.4 (P2)** 스트리밍 ingest.

### 8.2 지식그래프 모델링·온톨로지
- **FR-2.1 (P0)** 노드/엣지 스키마 + 제약·인덱스 (`graph/01_schema.cypher`).
  - AC: 유니크 제약·인덱스 생성, 스키마 검증 쿼리 통과.
- **FR-2.2 (P1)** 온톨로지 정의 문서화·버전관리 + (옵션) SHACL 무결성 게이트.
  - AC: 온톨로지 변경이 마이그레이션으로 추적됨.

### 8.3 탐지 엔진 (룰 + 그래프 알고리즘 + ML)
- **FR-3.1 (P0)** 공유 엔티티 탐지(주소/계좌/전화/차량). AC: 공유 고객 쌍/군집 반환 (Q1·Q3).
- **FR-3.2 (P0)** 엔티티 핫스팟(병원·정비소·계좌 청구 집중). AC: 임계치 이상 엔티티 반환 (Q2).
- **FR-3.3 (P0)** crash-for-cash 순환(교차 목격) 탐지. AC: 교차 목격 순환 고객 반환 (Q4 쿼리).
- **FR-3.4 (P1)** GDS 군집(WCC·Louvain) + 중심성(Degree·PageRank). AC: 주입 링이 단일 커뮤니티, 허브 상위 랭크.
- **FR-3.5 (P0)** 룰·그래프 신호 가중합 리스크 스코어 + 임계치 알림. AC: 정상 0점, 링 멤버 고점 분리 (Q5).
- **FR-3.6 (P2)** 그래프 임베딩/GNN 신종 패턴(설명가능성 보완 전제).

### 8.4 케이스 관리
- **FR-4.1 (P0)** 의심 케이스 큐(점수순) + 상태(미배정/조사중/사기/정상/보류). AC: 상태 전이·담당자 배정·이력.
- **FR-4.2 (P1)** 케이스별 연관 엔티티·청구·경로 자동 첨부.
- **FR-4.3 (P1)** 조사관 판정 라벨 기록(피드백 입력).

### 8.5 설명가능성 — 차별화 핵심
- **FR-5.1 (P0)** 탐지 근거 = **의심 관계 경로**를 명시적으로 반환(어떤 공유/순환이 점수에 기여했는지).
  - AC: 각 알림에 기여 신호·경로 목록 첨부.
- **FR-5.2 (P1)** 경로를 **자연어 근거문**으로 생성(LLM + 온톨로지 컨텍스트, provider 추상화).
  - AC: "R1·R2·R3은 동일 계좌·정비소를 공유하며 서로의 사고를 교차 목격함" 형태 소명문 생성. **환각 가드: 실재 경로에 없는 엔티티 인용 금지(검증 테스트).**

### 8.6 시각화·탐색
- **FR-6.1 (P0)** 케이스 관계망 뷰(노드·엣지, 의심 경로 하이라이트). AC: 링 서브네트워크 시각 분리.
- **FR-6.2 (P1)** 인터랙티브 확장(이웃 펼치기·필터·시간축).

### 8.7 피드백 루프·학습
- **FR-7.1 (P1)** 판정 라벨 누적 → 룰/임계치/모델 재튜닝 입력.
- **FR-7.2 (P2)** 라벨 기반 지도학습 재학습 파이프라인.

### 8.8 API·통합
- **FR-8.1 (P0/PoC=FastAPI)** REST API: 케이스 조회/점수/그래프 탐색 (OpenAPI). V1에서 NestJS BFF.
  - AC: OpenAPI 스펙 + 인증/인가.
- **FR-8.2 (P2)** 기간계 FDS 연동.

### 8.9 알림·리포팅
- **FR-9.1 (P1)** 임계치 초과 케이스 알림.
- **FR-9.2 (P1)** 경영 대시보드(적발률·절감액·처리량 KPI).

## 11. 기술 아키텍처 (v0.2 확정)
```
[소스: 청구/계약/차량/계좌/의료/정비 (PoC=합성)]
        │ (배치 ingest, 엔티티 해소, 가명처리)        ── ingest/ (Python)
        ▼
[Neo4j 지식그래프 + GDS] ──(WCC/Louvain/PageRank/Degree)──► [탐지·스코어링]  ── detection/
        │                                                        │
        │                                              [설명 레이어: LLM provider 추상화 + 경로 grounding]  ── explain/
        ▼                                                        ▼
[FastAPI REST] ───────────────────────► [React 콘솔 + vis-network 관계망]  ── api/ , console/
   (PoC; V1에서 NestJS BFF 추가)              └─ 케이스 관리 / 판정 피드백
[RBAC + 감사 미들웨어]  ── core/security/ (flux-platform 패턴 벤더링)
```
**스택:** Python 3.10+ / FastAPI / **Neo4j 5 + GDS** / PostgreSQL(케이스 메타, 옵션 PoC=SQLite) / Redis(옵션) / React 18 + Vite + vis-network / LLM(anthropic|openai|ollama 추상화).
**벤더링 출처:** flux-platform(`core/kg/algorithms`, `core/kg/rbac`, `domains/maritime` 플러그인 패턴) · flux-ontology-local(`llm.service`, `VisNetworkGraph`).
