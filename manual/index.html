보험 사기를
두 가지 방법으로 잡는 시스템

THOTH-ON은 여러 명이 짜고 치는 조직 사기와 혼자 부풀리는 개인 사기, 두 종류를 각각 다른 방법으로 탐지합니다. 왜 의심스러운지 조사관이 바로 이해할 수 있는 한국어 설명도 자동으로 만들어 줍니다.

개인형 — 속성 AI 탐지

01

한 문장 소개

THOTH-ON이란?
THOTH-ON은 보험금을 노린 '짜고 친' 조직 사기와 혼자 부풀리는 개인 과장 청구, 두 종류의 사기를 각각 다른 방법으로 찾아내고, 왜 의심스러운지 한국어로 설명해 주는 시스템입니다.

기존 보험 심사 시스템은 청구 한 건씩 따로따로 검토합니다. 그러다 보니 여러 사람이 짜고 친 조직 사기는 건건이 보면 평범해 보여서 그냥 넘어가기 쉽습니다. 개인이 혼자 부풀리는 단발 과장 청구는 흔적이 더욱 적습니다.

THOTH-ON은 관계망(그래프) 레이어로 조직형 공모를 잡고, 속성 AI 레이어로 개인형 과장 청구를 잡아 — 두 레이어를 합쳐 전체 사기의 94%까지 탐지합니다(시범 데이터 기준). 조사관은 "왜 이 케이스가 의심스러운가"를 시스템이 만들어 준 한국어 설명으로 바로 파악하고 판단을 내릴 수 있습니다.

이 안내서는 누구를 위한 것인가요?
보험사 임원, 조사 담당자(SIU), 의사결정자 — IT 전문 지식 없이도 읽을 수 있도록 쉬운 말로 썼습니다. 기술 설치 안내는 맨 뒤 섹션 10에 따로 정리했습니다.
02

어떤 문제를 푸나요?

보험 사기는 점점 조직화되고 있습니다. 한 사람이 혼자 허위 청구를 넣는 시대는 지났습니다. 요즘 사기는 여러 사람이 역할을 나눠, 서로를 도와 가며 반복적으로 사고를 꾸밉니다.

🚗
고의 교통사고 (crash-for-cash)
보험금을 노리고 일부러 사고를 냅니다. 가담자들이 번갈아 가며 사고를 일으키고 서로의 목격자가 되어 줍니다.
고위험
🔧
가짜 수리비 청구
실제로 수리하지 않거나 과장된 수리비를 특정 정비소와 짜고 청구합니다. 특정 정비소에 청구가 비정상적으로 몰립니다.
중위험
🏦
계좌 공유 조직
여러 사람이 한 계좌로 보험금을 받습니다. 따로따로 청구해도 돈이 모이는 계좌가 같으면 공모 조직일 가능성이 높습니다.
고위험

왜 기존 방법으로는 못 잡나요?

사건을 하나씩 따로 보면 각각은 정상처럼 보입니다. 금액도 평범하고, 사고 경위도 자연스럽습니다. 그래서 지금까지는 이런 사기를 놓쳐 왔습니다.

💡
비유: 흩어진 점만 보면 아무 그림도 보이지 않습니다. 하지만 그 점들을 선으로 이으면 그림이 드러납니다. THOTH-ON은 보험 청구·사람·계좌·정비소를 점으로, 그 사이의 관계를 선으로 이어 '조직 사기 그림'을 그려 냅니다.

THOTH-ON은 수천 건의 청구 데이터 전체를 하나의 거대한 관계 지도로 만들어 "이 사람들이 서로 어떻게 엮여 있는가"를 한눈에 파악합니다. 이 방식은 개별 건 심사로는 절대 볼 수 없는 패턴을 보여 줍니다.

03

어떻게 찾아내나요?

THOTH-ON은 세 가지 핵심 신호를 탐지합니다. 신호가 하나라도 감지되면 점수가 올라가고, 여러 신호가 겹칠수록 위험 점수는 더 높아집니다.

🔗
신호 ① 계좌·전화 공유
서로 다른 사람이 같은 계좌나 전화번호를 공유하면 의심스럽습니다. 가족도 아닌데 보험금 입금 계좌가 같다면 공모 조직일 가능성이 높습니다.
강한 신호
🏪
신호 ② 특정 장소 집중
같은 정비소나 병원에 청구가 비정상적으로 몰립니다. 짜고 친 협력 업체일 가능성을 나타냅니다.
중간 신호
👥
신호 ③ 번갈아 목격
A가 B의 사고를 목격하고, B가 A의 사고를 목격합니다. 실제 우연의 일치라기에는 너무 규칙적입니다. 공모자끼리 서로 보증인이 되어 주는 패턴입니다.
강한 신호

위험 점수 (0~100점)

위 신호들의 개수와 강도를 합산해 위험 점수를 계산합니다. 점수가 높을수록 먼저 조사해야 할 사건입니다.

🗺️
지식그래프란? 사람·사건·통장·차·정비소를 점(노드)으로 표시하고, 그 사이의 관계를 선(엣지)으로 이은 거대한 관계 지도입니다. 이 지도를 컴퓨터가 분석하면 "이 사람들이 한 조직"임을 자동으로 알아낼 수 있습니다.

설명 가능한 AI — 근거 있는 말만 합니다

점수만 내고 끝내지 않습니다. THOTH-ON은 "왜 이 케이스가 위험한가"를 관계 지도에서 실제로 확인된 사실만을 근거로 한국어 문장으로 설명합니다. AI가 없는 사실을 지어내지 못하도록 환각가드(HallucinGuard)라는 안전장치가 모든 설명문을 검증합니다.

핵심

두 종류의 사기, 두 개의 탐지 레이어

왜 두 가지 방법이 필요한가?
보험 사기는 크게 두 종류입니다. 조직형(여러 명이 짜고 치는 공모)과 개인형(혼자 청구를 부풀리는 단발 과장). 현실에서는 개인형이 더 많습니다 — 실데이터 기준 사기의 약 76%가 개인형, 24%가 조직형입니다. 두 종류는 성격이 달라서, 한 가지 방법으로는 둘 다 잡을 수 없습니다.

THOTH-ON의 답: 듀얼 레이어

THOTH-ON은 두 종류의 사기를 각각 다른 엔진으로 탐지하고, 두 결과를 합쳐 최종 위험 점수를 냅니다.

🔍
비유: 조직형은 "점들을 잇는 선의 모양"으로 판단하고, 개인형은 "점 하나의 특징"으로 판단합니다. 두 판단을 합쳐야 빠짐 없이 잡을 수 있습니다.

레이어 1 — 관계망(그래프) 탐지: 조직형에 강력

사람·계좌·정비소·사건을 점으로 이어 관계 지도를 만들고, 수상한 연결 패턴(계좌 공유, 교차 목격, 정비소 집중)을 발견합니다. 조직형 사기는 이 방법으로 재현율 0.98까지 잡히며, 헛알림이 적어 자동 알림을 믿을 수 있습니다.

단 개인형은 혼자 청구를 부풀릴 뿐 공모 네트워크가 없어, 이 레이어로는 개인형 사기의 약 1%만 탐지됩니다.

레이어 2 — 속성 AI 탐지: 개인형에 강력

청구 하나하나의 "특징"을 AI가 분석합니다. 본인과실 여부, 주소 변경 직후 청구, 자기부담금, 사고 지역(농촌·도시) 등 수십 가지 신호를 종합해 각 청구의 위험도를 계산합니다.

쓸수록 똑똑해집니다 — 피드백 루프

보통의 탐지 시스템은 한 번 만들어 두면 그대로 멈춰 있습니다. THOTH-ON은 다릅니다. 조사관이 내린 '사기 확정 / 정상' 판정이 곧 시스템의 새로운 학습 교재가 됩니다. 조사관이 사건을 처리할수록 시스템은 자기 회사의 사기 패턴을 더 잘 알아보게 됩니다.

🎓
신입 조사관이 선배의 판정을 보며 감을 익히는 것과 같습니다. THOTH-ON은 여러분 회사 조사관들의 판정을 모아 학습하기 때문에, 외부에서 만든 일반 모델보다 우리 회사의 실제 사기 유형에 점점 더 맞춰집니다.

한 바퀴 도는 학습 고리

  1. 1
    조사관이 판정합니다 사건 상세 화면에서 '사기 확정 · 정상 · 보류' 버튼을 누릅니다. 이 판정이 시스템에 정답 라벨로 기록됩니다.
  2. 2
    판정이 운영 라벨로 쌓입니다 처음에 들어온 합성·시범 라벨이 아니라, 실제 조사관이 검증한 결과가 별도로 누적됩니다. 우리 회사만의 진짜 정답지가 만들어집니다.
  3. 3
    모델이 다시 학습합니다 버튼 한 번(또는 정기 일괄 실행)으로 그동안 쌓인 판정을 반영해 탐지 모델을 재학습합니다. 새로 배운 모델은 안전하게 보관(영속화)됩니다.
  4. 4
    새 모델이 곧바로 점수 매기기에 투입됩니다 재학습된 모델은 다음 사건들의 위험 점수를 매기는 데 실제로 사용됩니다. '학습만 하고 안 쓰는' 모델이 아니라 현장에 바로 반영됩니다.
  5. 5
    의심 사건 목록이 자동으로 재정렬됩니다 큐의 점수가 새 모델 기준으로 다시 계산되어, 조사관은 다음 출근 때 한층 정교해진 우선순위로 사건을 받습니다. 그리고 다시 1번으로 — 고리가 한 바퀴 돕니다.
정직한 성능 표기 원칙. 처음의 '시범 라벨 기준 성능'과 '조사관 판정 반영 후 성능'은 서로 다른 정답지로 측정되므로 두 수치를 단순 비교하지 않습니다. 시스템은 두 기준을 나란히 보여 주고, 변화량(delta)은 어디까지나 참고치임을 항상 함께 표기합니다 — 좋아 보이게 부풀리지 않습니다.
기술 담당자 참고. 이 고리는 REST API로 노출됩니다 — POST /cases/{id}/verdict(판정 기록) · POST /detection/retrain(판정 반영 재학습) · POST /detection/rescore(큐 재스코어링) · GET /detection/model(활성 모델 메타 조회). 모든 접근은 RBAC·감사 기록 대상입니다.

점수 분리도 진짜 의심 사건과 일반 사건의 점수가 얼마나 멀리 떨어져 있는가를 나타냅니다. 이 값이 높을수록 시스템이 두 집단을 잘 구별하고 있다는 뜻입니다.
📊
상태 분포 막대 차트 미배정 / 조사중 / 사기 확정 / 정상 / 보류 건수를 막대로 보여 줍니다. 미배정이 많으면 조사 인력이 더 필요하다는 신호입니다.
💰
일일 처리량 추정 / 절감액 추정 조사관 1명이 하루 처리할 수 있는 건수 추정치와, 사기 확정 시 절감되는 금액 추정치입니다. 합성 데이터 PoC 기준의 추정치이며 실제 운영 수치를 보장하지 않습니다.
08

실제로 이렇게 씁니다

보험사 조사관 김 대리가 아침에 시스템을 열었습니다. 이후 흐름을 따라가 봅니다.

목록을 열어 점수 높은 사건을 고릅니다
김 대리는 오늘 처리할 사건을 찾기 위해 '미배정' 필터를 켭니다. 점수 100점짜리 사건 여러 건이 보입니다. 가장 위에 있는 CASE-CUST-05001을 클릭합니다.
한국어 설명문으로 왜 의심인지 바로 파악합니다
오른쪽 패널에 설명문이 펼쳐집니다. "고객 CUST-05001은 고객 CUST-05010, CUST-05011과 동일 계좌를 공유하며, 서로의 사고를 번갈아 목격한 기록이 있습니다." — 5초 만에 의심 이유를 파악했습니다.
관계 그림으로 엮인 무리를 확인합니다
'관계망' 탭을 누릅니다. CUST-05001을 중심으로 여러 빨간 마름모가 같은 계좌 점으로 모여 있는 그림이 보입니다. 한 조직임이 한눈에 드러납니다.
'사기 확정' 또는 '보류' 버튼을 누릅니다
설명문과 그림을 검토한 뒤 충분한 근거가 있다고 판단하면 '사기 확정'을 누릅니다. 더 확인이 필요하면 '보류'를 선택하고 메모를 남깁니다.
판정이 쌓여 시스템이 더 똑똑해집니다
김 대리의 판정 결과는 현황판에 반영됩니다. 사기 확정 / 정상 처리 레이블이 쌓일수록 시스템의 탐지 정확도가 높아집니다.
09

무엇이 다른가요?

경영진이 가장 많이 묻는 질문입니다. "기존 시스템과 무슨 차이가 있나요?"

🔒
인터넷 없이 작동합니다
보험사 내부 전용망에서 완전히 독립 작동합니다. 고객 개인정보가 외부 클라우드나 인터넷으로 나가지 않습니다. 금융 규제 환경에서 안심하고 쓸 수 있습니다.
💬
"왜 의심인지"를 사람이 이해하는 말로 설명합니다
숫자 점수만 내놓지 않습니다. 어떤 관계가 얽혀 있고, 무슨 패턴이 발견됐는지 한국어 문장으로 설명해 줍니다. 조사·소명서 작성이 빠릅니다.
🕸️
관계망 전체를 한꺼번에 봅니다
개별 청구 건 하나씩 심사하는 방식에서 벗어나, 수천 건의 청구 데이터를 한 장의 관계 지도로 만들어 조직 전체를 발견합니다.
🛡️
개인정보를 암호로 가려 저장합니다
이름·주민번호·계좌번호는 시스템에 저장될 때 SHA-256 방식으로 암호화됩니다. 화면에는 짧은 암호 코드만 표시됩니다.
📋
누가 무엇을 봤는지 모두 기록합니다
케이스 조회, 판정 변경 등 모든 행동이 감사 기록에 남습니다. 나중에 "누가 언제 이 사건을 보고 어떤 판정을 내렸는가"를 추적할 수 있습니다.
🤖
AI가 지어낸 말을 걸러 냅니다
환각가드(HallucinGuard)가 AI 설명문을 검증합니다. 실제 데이터에 없는 내용을 AI가 꾸며냈다면 그 설명은 화면에 표시되지 않습니다.
🟢
지금 어떤 AI가 돌고 있는지 한눈에 보입니다
콘솔 상단의 상태 배지로 현재 활성화된 AI 엔진(예: 내부망 Ollama)을 표시합니다. 외부 클라우드 AI를 쓰지 않고 사내 모델로만 돌고 있음을 운영자가 즉시 확인할 수 있습니다(GET /llm/status).
♻️
조사관 판정으로 스스로 학습합니다
조사관이 내린 '사기 확정 / 정상' 판정을 모아 탐지 모델을 재학습하고, 새 모델을 곧바로 점수 매기기에 반영합니다. 쓸수록 우리 회사 사기 패턴에 맞춰집니다(피드백 루프 참고).
성능 수치 안내: 탐지율·F1 등의 수치는 현재 합성 시범 데이터 기준으로 측정된 것입니다. 단, 속성 AI(개인형 탐지)의 ROC-AUC 0.82는 실제 공개 자동차보험 데이터(캐글, 15,420건)로 별도 검증한 수치입니다. 실제 보험사 데이터 환경에서의 전체 성능은 NDA 파일럿 단계에서 재측정이 필요합니다.
09+

한국 실제 사기 수법 5종 — THOTH-ON이 잡는 패턴

THOTH-ON의 탐지 규칙은 이론이 아닌 금융감독원·KIRI(보험연구원) 실제 적발 사례를 기반으로 설계됐습니다. 현장에서 반복적으로 나타나는 5가지 수법을 시스템에 반영했습니다.

🏥
허위입원 조직형
실제로 입원하지 않았거나 며칠만 입원하고 장기 입원으로 청구합니다. 여러 명이 같은 병원을 돌아가며 이용하고, 담당 의사·직원과 짜고 진단서를 허위로 발급받습니다.
그래프 탐지 · 병원 허브
💥
고의충돌 공모
보험금을 노리고 두 차량이 서로 합의 하에 충돌합니다(crash-for-cash). 가담자들이 번갈아 사고를 일으키고, 서로의 사고 목격자가 되어 주는 패턴이 나타납니다.
그래프 탐지 · 교차목격
🔧
정비비 과다청구
특정 정비소와 짜고, 실제로 수리하지 않은 항목이나 부품을 청구서에 추가합니다. 특정 정비소에 청구가 비정상적으로 집중되는 패턴으로 탐지됩니다.
그래프 탐지 · 정비소 허브
🚘
운전자 바꿔치기
사고 당시 보험에 가입되지 않은 운전자가 실제 운전했지만, 기명 피보험자가 운전한 것처럼 진술합니다. 관계 그래프에서 차량·사람의 연결 패턴이 어색하게 나타납니다.
GDS 임베딩 탐지
🤝
보험설계사 개입
보험설계사가 고객과 짜고 허위·과장 청구를 유도하거나 직접 사기에 가담합니다. 특정 설계사를 통한 청구에 사기율이 집중되는 패턴으로 감지됩니다.
속성 AI 탐지
수법별 탐지 방식이 다릅니다. 조직이 움직이는 수법(허위입원·고의충돌·정비비·운전자 교체)은 주로 관계망(그래프) 레이어로, 개인 단위의 과장 청구나 설계사 개입 의심은 속성 AI 레이어로 탐지합니다. 두 레이어를 함께 쓰기 때문에 수법이 달라도 하나의 시스템에서 탐지됩니다.

수법 유형별 탐지 성능 (합성 데이터 기준)

설치 안내 — 기술 담당자용

이 섹션은 IT 기술 담당자용입니다. 실제 시스템 설치·실행을 담당하는 분이 아니라면 이 부분은 건너뛰셔도 됩니다.

사전 요구사항: Python 3.10+, Docker Desktop (GDS 플러그인 포함), Node.js 18+, make

1 — 환경 변수 설정
cp .env.example .env
# .env 파일을 열어 NEO4J_PASSWORD 등을 설정합니다
2 — Python 환경 설치
make install
# Python 가상환경 생성 + 의존성 패키지 설치
3 — 데이터베이스 기동
make up
# Neo4j 5 + GDS 컨테이너 기동 (Docker)
4 — 스키마 초기화 + 시범 데이터 생성·적재
make schema   # 제약 조건·인덱스 생성
make synth    # 합성 시범 데이터 생성 (사기 조직 패턴 포함)
make seed     # 데이터베이스에 적재 (가명처리 포함)
5 — API 서버 기동
make serve
# FastAPI 서버 → http://localhost:8468/docs (OpenAPI UI)
6 — 콘솔 기동
cd console && npm run dev
# React 개발 서버 → http://localhost:5847
